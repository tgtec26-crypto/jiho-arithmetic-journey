# 1. 문제 정의 (Problem Statement)
현재 "지호의 산수 여행" 프로젝트는 단일 `index.html` 파일에 모든 게임 로직(8종), 스타일(CSS), UI 구조(HTML)가 섞여 있어 유지보수가 매우 어려운 모놀리식 구조입니다. 

**주요 과제**:
- **코드 복잡성 해결**: 기능을 독립된 Svelte 컴포넌트로 분리하여 가독성과 재사용성을 높여야 합니다.
- **상태 관리 통합**: 여러 개로 분산된 전역 상태를 Svelte Store로 통합하여 데이터 흐름을 명확히 해야 합니다.
- **현대적 UI 구현**: 인라인 스타일과 전역 CSS 대신 Tailwind CSS를 도입하여 디자인 시스템을 체계화합니다.
- **PWA 안정성 확보**: 수동으로 관리하던 Service Worker를 `vite-plugin-pwa`를 통해 자동화하여 오프라인 지원 품질을 개선합니다.

# 2. 요구사항 (Requirements)
## 기능적 요구사항
- **8종의 미니 게임 이식**: 덧셈, 뺄셈, 거미 사다리, 시계 보기 등 모든 게임 로직을 Svelte 컴포넌트로 분리하여 동일하게 구현합니다.
- **통합 상태 관리**: 전체 점수, 현재 게임 진행 상태, 사용자 설정을 Svelte Store(`gameStore.ts`)를 통해 전역적으로 관리합니다.
- **PWA 오프라인 지원**: 모든 정적 자산(JS, CSS, 이미지, 폰트)을 캐싱하여 인터넷 연결 없이도 모든 게임을 플레이할 수 있어야 합니다.
- **동적 피드백**: `canvas-confetti` 라이브러리를 사용해 정답 시 시각적 보상을 제공하고, 인터랙티브한 UI 효과를 유지합니다.

## 비기능적 요구사항
- **최적화된 렌더링**: 모바일 및 저사양 기기에서도 끊김 없는 애니메이션과 빠른 로직 처리를 목표로 합니다.
- **로직 안정성 확보**: 산술 연산 및 게임 상태 업데이트 로직에 대해 Vitest를 사용한 단위 테스트(Unit Test)를 작성합니다.
- **현대적 UI 구현**: Tailwind CSS를 사용하여 모든 스타일을 Scoped CSS 또는 유틸리티 클래스로 재구성합니다.
- **TypeScript 도입**: 모든 게임 데이터 모델과 상태 변화 함수에 타입을 정의하여 코드의 안정성을 높입니다.

## 제약 사항
- **클라이언트 전용**: 데이터 저장(DB)이나 로그인 기능을 배제한 순수 클라이언트 앱으로 유지합니다.
- **한국어 전용**: 현재 상태를 유지하며 다국어(i18n) 처리는 범위에서 제외합니다.
- **배포 설정 유지**: 기존의 `vercel.json` 설정을 유지하며 `@sveltejs/adapter-static`을 사용해 정적 사이트로 배포합니다.

# 3. 접근 방식 (Approach)
## 선택된 접근 방식: 컴포넌트 중심 아키텍처 (Component-Centric Architecture)
- **Svelte Store 기반 상태 관리**: `gameStore.ts`에서 점수, 현재 게임 모드, 진행률을 중앙 집중식으로 관리합니다. — *각 게임 간의 데이터 공유를 직관적으로 처리하기 위해 도입 (요구사항: 통합 상태 관리)*
- **동적 컴포넌트 스위칭**: 단일 페이지 내에서 `{#if}` 블록 또는 `<svelte:component>`를 사용해 각 미니 게임(덧셈, 거미 사다리 등)을 교체합니다. — *기존의 부드러운 게임 전환 경험을 유지하면서 독립적인 로직 분리를 위해 결정 (요구사항: 1:1 이식)*
- **Tailwind CSS 디자인 현대화**: 기존의 복잡한 CSS 코드를 Tailwind CSS 클래스로 재구축합니다. — *가독성 높은 스타일 관리와 빠른 UI 개발을 위해 선택 (요구사항: 현대적 UI 구현)*
- **Vite PWA 플러그인 도입**: `vite-plugin-pwa`를 사용해 캐싱 전략을 자동화합니다. — *안정적인 오프라인 지원과 빌드 결과물 캐싱을 위해 도입 (요구사항: PWA 오프라인 지원)*

## 고려된 대안 및 의사 결정 매트릭스
- **대안 1: 라우트 기반 아키텍처** (각 게임별 개별 페이지 구성): 미래 확장성은 높지만, 현재 규모에서는 과하고 전환 시의 사용자 경험이 매끄럽지 않아 배제되었습니다.

# 4. 아키텍처 (Architecture)
## 컴포넌트 계층 구조 (Component Hierarchy)
- `App.svelte` (메인 엔트리): 전체 레이아웃과 게임 전환 상태를 감시합니다. — *메인 페이지의 상태(state)에 따라 적절한 컴포넌트를 호출하기 위함 (요구사항: 동적 컴포넌트 스위칭)*
  - `Header.svelte`: 로고, 전체 점수, 뒤로 가기 버튼을 포함합니다.
  - `GameContainer.svelte`: 현재 선택된 게임 컴포넌트를 렌더링하는 컨테이너입니다. — *중앙 로직 분리를 위해 구현 (요구사항: 8종의 미니 게임 이식)*
    - `Menu.svelte`: 8개 게임 선택 카드 목록(4열 그리드 유지).
    - `ArithmeticGame.svelte`: 덧셈/뺄셈/곱셈/나눗셈 공통 게임 로직 처리. — *중복 로직 제거를 위한 공통 컴포넌트 설계 (요구사항: 1:1 이식)*
    - `SpiderGame.svelte`, `ClockGame.svelte`, `CompareGame.svelte`, `MoneyGame.svelte`: 각 게임별 고유 인터랙티브 로직.
  - `ProgressBar.svelte`: 현재 게임 세션의 진행률 표시.
  - `Confetti.svelte`: 정답 시 폭죽 효과 실행 트리거. — *시각적 보상을 통한 학습 동기 부여를 위해 독립 컴포넌트로 분리 (요구사항: 동적 피드백)*

## 데이터 흐름 (Data Flow)
- **중앙 집중식 상태 관리**: `gameStore.ts`는 모든 컴포넌트에서 읽고 쓸 수 있는 단일 소스 역할을 합니다. — *전역 점수와 진행률을 일관되게 유지하기 위함 (요구사항: 통합 상태 관리)*
- **반응형 업데이트**: 게임 컴포넌트에서 정답을 맞히면 `gameStore`의 점수를 업데이트하고, `Header`와 `ProgressBar`는 이를 즉시 반영하여 화면에 표시합니다. — *Svelte의 반응형 선언을 통해 UI와 로직을 동기화하기 위함 (요구사항: 최적화된 렌더링)*

# 5. 에이전트 팀 (Agent Team)
- **architect**: SvelteKit 프로젝트의 전체 폴더 구조를 생성하고, 컴포넌트 간의 의존성 및 데이터 흐름(Svelte Store)을 최종 결정합니다. — *프로젝트 초기화와 공통 아키텍처 수립을 위해 결정 (요구사항: 통합 상태 관리)*
- **coder**: 기존 `index.html`의 8개 게임 로직을 Svelte 컴포넌트로 이식하고, TypeScript 타입을 적용합니다. — *대량의 로직 이식을 정확하게 처리하기 위해 결정 (요구사항: 8종의 미니 게임 이식)*
- **design_system_engineer**: 기존의 CSS 코드를 Tailwind CSS로 변환하고, 반응형 레이아웃과 애니메이션 품질을 개선합니다. — *Tailwind CSS의 효율적인 클래스 구성과 현대적 UI 구현을 위해 결정 (요구사항: 현대적 UI 구현)*
- **tester**: 각 게임의 핵심 산술 연산 및 상태 업데이트 로직에 대해 Vitest 단위 테스트를 작성하고 실행합니다. — *게임 로직의 무결성을 보장하기 위해 결정 (요구사항: 로직 안정성 확보)*

# 6. 위험 평가 (Risk Assessment)
- **개별 게임 구현의 복잡성**: '거미 사다리'와 같이 드래그 앤 드롭 및 복잡한 시각 효과가 포함된 게임은 단순 이식이 어려울 수 있습니다. — *기존의 수동 이벤트 핸들러(mousedown, touchmove 등)를 Svelte의 선언적 로직으로 변환 시 로직 오류 발생 가능성 대비 (요구사항: 1:1 이식)*
  - **대응**: 복잡한 게임의 핵심 로직을 Vitest 단위 테스트로 미리 추출하여 검증한 뒤 Svelte 컴포넌트로 이식합니다.
- **PWA 및 Service Worker 구성 오류**: `vite-plugin-pwa` 설정이 기존의 수동 `sw.js` 캐싱 전략과 충돌하거나 정적 자산 캐싱이 누락될 수 있습니다. — *정적 어댑터와 PWA 설정 간의 동기화 문제 방지 (요구사항: PWA 오프라인 지원)*
  - **대응**: `vite-plugin-pwa`의 `injectRegister` 옵션과 빌드 설정을 단계별로 테스트하여 로컬 및 Vercel 배포 후 캐싱 동작을 철저히 검증합니다.
- **Tailwind CSS 스타일 충돌 및 누락**: 기존의 전역 CSS 변수(`--deep-blue` 등)를 Tailwind 클래스로 변환하는 과정에서 특정 기기의 레이아웃이 깨질 위험이 있습니다. — *아이들이 사용하는 다양한 해상도의 기기에 대응하기 위함 (요구사항: 현대적 UI 구현)*
  - **대응**: 핵심 레이아웃에 대해 모바일 우선(Mobile-first) 접근 방식을 적용하고, 주요 브라우저 및 해상도에서 수동 검증을 병행합니다.
- **데이터 유실 위험**: 백엔드가 없으므로 페이지 새로고침 시 모든 게임 상태와 점수가 초기화될 수 있습니다. — *아이들의 학습 성취감을 유지하기 위해 상태 보존이 필요함 (요구사항: 통합 상태 관리)*
  - **대응**: `localStorage`와 연동되는 Svelte Store(영속성 스토어)를 구현하여, 새로고침 후에도 이전 점수와 진행 상태를 복구하도록 설계합니다.

# 7. 성공 기준 (Success Criteria)
- **기능 일치성 (8/8)**: 기존 `index.html`에 있던 8가지 게임이 SvelteKit 컴포넌트로 완벽하게 이식되어 동일한 기능을 수행해야 합니다.
- **디자인 현대화**: 모든 CSS 코드가 삭제되고 Tailwind CSS의 유틸리티 클래스와 Scoped CSS로 완전히 대체되어야 합니다.
- **상태 관리 통합**: 전체 점수, 진행률, 현재 게임 상태가 Svelte Store(`gameStore.ts`)를 통해서만 업데이트되고 관리되어야 합니다.
- **안정적인 오프라인 지원**: 빌드 후 PWA로 설치하여 네트워크 연결이 없는 환경에서도 모든 게임이 정상적으로 실행되어야 합니다.
- **배포 성공**: Vercel에 `@sveltejs/adapter-static`을 사용해 성공적으로 배포되어야 하며, 기존의 `vercel.json` 설정이 정상적으로 작동해야 합니다.
- **테스트 통과**: Vitest를 통한 핵심 게임 로직 단위 테스트의 성공률이 100%여야 합니다.
